🏗️ Phase 1: The "Zero-Overhead" Core

Goal:
Fix the "Hard Stop" of data conversion. If it's not zero-copy, it's too slow.

🔹 Issue #1: Zero-Copy Mapping for U256 ↔ Fr Field Elements

Context:
Existing solutions often incur overhead during conversion. We need a no_std, trait-based approach to map Soroban's host-managed U256 to our ethnum-backed field elements without heap allocation.

Tasks:

Implement SafeFrom for Fr (Field Element) using constant-time reduction
Create a HostConvert trait to handle the env.crypto().bn254_fr_from_u256() host call
Ensure the implementation is #[inline(always)] to minimize WASM stack depth

Target:
~15% reduction in conversion latency compared to standard Rust implementations

🔹 Issue #2: CAP-0075 Poseidon2 Sponge Construction

Context:
Simply calling the host function isn't enough. Developers need a high-level "Sponge" API (absorb / squeeze) that doesn't blow the instruction budget.

Tasks:

Implement the Poseidon2 state machine
Directly interface with env.crypto_hazmat().poseidon2_permutation() to avoid guest-side loop overhead
Provide a hash_to_field utility compatible with Noir/Circom constraints
⚡ Phase 2: Resource & Gas Superiority

Goal:
Beat the 400M instruction limit and the 64KB WASM cap

🔹 Issue #3: Static Verification of Instruction Budgets

Context:
Soroban-ZK-Std should be the first library to provide "Gas-Guaranteed" primitives

Tasks:

Create a benchmarking suite to measure instruction usage for operations like G1Add and PairingCheck
Publish a GAS.md table so developers can estimate costs before writing code

Why this beats Tupui:
Shifts the library from "it works" → "predictable for production"

🔹 Issue #4: Dead-Code Elimination & WASM Bloat Audit

Context:
Many Rust ZK libraries pull in std or heavy formatting dependencies

Tasks:

Audit all dependencies (e.g., ethnum) to ensure zero-cost abstractions
Implement lightweight custom panic handlers to replace std::panic

Target:
Core math module size < 12 KB
